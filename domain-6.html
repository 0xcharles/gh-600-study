Governance, Guardrails,
& Accountability

Domain 6 is about safety and control — defining how much autonomy agents have, enforcing boundaries through GitHub-native controls, and maintaining audit trails so every action is accountable. This domain is the guardian of the entire agentic system.

1. Risk-Based Autonomy and Action Boundaries

The foundational concept of Domain 6 is that not all agent actions are equal. Some actions are low-risk (reading a file, creating a draft PR), while others are high-risk (deploying to production, modifying secrets, merging to main). The exam expects you to map risk levels to autonomy levels and design appropriate boundaries.

Autonomy Levels

Level Name Behavior Example Actions
L0 No Autonomy Agent only responds to explicit human commands; no independent action Copilot Chat completions, code explanations
L1 Supervised Agent proposes actions; human must approve before execution Creating PRs, suggesting code changes, opening issues
L2 Constrained Agent executes autonomously within defined boundaries; human can override Running tests, linting, creating branches, committing to feature branches
L3 Full Autonomy Agent executes all actions independently; human only notified of results Deploying to staging, merging non-critical PRs, triaging issues

Mapping Risk to Autonomy

The exam's core governance rule: Higher risk = Lower autonomy. Map each action to a risk level, then assign the appropriate autonomy level:

  • Low risk (reading files, searching code, creating draft PRs) → L2 or L3 autonomy
  • Medium risk (committing to branches, running CI, modifying non-critical configs) → L1 or L2 autonomy
  • High risk (merging to main, deploying to production, modifying secrets, changing access controls) → L0 or L1 autonomy (human must approve)

📘 Teaching Tip — The Autonomy Spectrum Is Continuous: Don't think of autonomy levels as fixed tiers. Think of them as a spectrum where you slide the autonomy dial based on risk. The exam expects you to calibrate — not just choose a level, but explain why that level is appropriate for the given risk profile. Always justify your autonomy choice with the specific risk involved.

2. GitHub-Native Controls: Rulesets, Checks, CODEOWNERS, Environments

GitHub provides a suite of built-in controls that serve as guardrails for agent behavior. The exam tests your knowledge of each control's purpose, configuration, and interaction.

Rulesets

Rulesets are the first line of defense. They define allowed and disallowed operations at the repository or organization level.

  • What they control: Branch creation, branch deletion, pushing to protected branches, tag creation, required linear history
  • Use for: Preventing agents from bypassing standard workflows (e.g., directly pushing to main)
  • Exam tip: Rulesets are enforced at the Git protocol level — even agents with write access cannot bypass them

Required Checks / Status Checks

Required checks enforce that certain CI workflows must pass before a PR can merge. These are defined in branch protection rules or rulesets.

  • What they control: Which workflows must pass (CodeQL, tests, linting, build)
  • Use for: Ensuring agent output meets quality gates before entering main
  • Exam tip: Required checks are the bridge between Domain 4 (evaluation) and Domain 6 (guardrails)

CODEOWNERS

CODEOWNERS automatically assigns reviewers based on file paths. This ensures that changes to critical files are reviewed by the right people or teams.

  • What it controls: Who must review changes to specific files or directories
  • Use for: Ensuring agent changes to sensitive files (config, security, deployment scripts) get expert review
  • Exam tip: CODEOWNERS is not just about humans — an agent can be a CODEOWNER too (agent profile can be listed)

Environments

GitHub Environments add a layer of protection around deployment targets. Each environment can have its own protection rules.

  • What they control: Who or what can deploy to an environment; required approvals; wait timers
  • Use for: Protecting production deployments — require human approval even if an agent initiates the deployment
  • Exam tip: Environment protection rules can require multiple approvers and enforce wait times between deployment stages
# Example: CODEOWNERS for sensitive agent-controlled files # All agent configuration files need senior dev review .github/agents/ @senior-devs # Deployment scripts need ops review deploy/ @ops-team # Security configs need security team review security/ @security-team # Root-level config files need both *.yml *.yaml @senior-devs @ops-team

⚠️ Exam Trap: A common exam question presents a scenario where an agent bypasses a protection mechanism and asks: "Which control should have prevented this?" If an agent pushes directly to main → rulesets/branch protection. If an agent merges without passing tests → required status checks. If an agent deploys to production without approval → environment protection rules. Know which control addresses which type of bypass.

3. Guardrails: Layered Defense in Depth

Governance works best as a layered defense. No single guardrail is sufficient — you need multiple layers that reinforce each other. The diagram below shows the six layers of defense, from outermost (broadest protection) to innermost (most specific):

Figure 6.1 — Guardrails Layered Defense: Six layers from outermost (broad branch protection) to innermost (specific pre-receive hooks). Each layer adds additional protection; an agent must pass through all layers to complete a high-risk action.

The key principle: defense in depth. No single guardrail is sufficient. If an agent bypasses branch protection (Layer 1), rulesets (Layer 2) still enforce Git protocol rules. If an agent somehow creates a valid PR, environment protection (Layer 3) still requires approval for deployment. Each layer adds friction specifically calibrated to the risk level of the action.

🎓 Teaching Tip — Memorize the Layer Order: The exam WILL ask about the layering order. Remember the acronym BREAD-H: Branch Protection → Rulesets → Environment Protection → Approvals → Data/Audit Logs → Hooks. This is the order from outermost to innermost. If a question asks which layer catches a specific bypass, trace from the outside inward to find the first layer that would have caught it.

4. Human-in-the-Loop Workflows for High-Risk Actions

Some actions are too risky for even the most constrained agent. The exam expects you to know when and how to insert a human-in-the-loop (HITL) gate into your agent workflows.

When to Require Human Approval

  • Deploying to production — requires environment approval rules
  • Merging to main or protected branches — requires PR review approval
  • Modifying security configurations — requires security team sign-off
  • Making changes to compliance-scoped files — CODEOWNERS review required
  • Actions that incur cost — cloud resource provisioning, API usage at scale

HITL Implementation Patterns

  • PR review required: Agent creates a PR with changes; human must review and approve before merging
  • Environment approval gates: Even if an agent triggers a deployment workflow, the environment protection rule requires a human approver
  • Workflow dispatch with manual trigger: High-risk workflows require a human to manually trigger them via workflow_dispatch with required inputs
  • Deployment protection rules with wait timer: Forces a cool-down period before deployment, giving humans time to intervene
  • Required reviewers in CODEOWNERS: Critical files automatically get human reviewers added to the PR
# Example: Human-in-the-loop deployment workflow name: Deploy with Approval on: workflow_dispatch: inputs: environment: description: 'Target environment' required: true type: environment jobs: deploy: runs-on: ubuntu-latest environment: name: ${{ github.event.inputs.environment }} # Environment protection rules require manual approval # configured in Settings > Environments steps: - uses: actions/checkout@v4 - name: Deploy run: ./deploy.sh

⚠️ Exam Trap: The exam will test whether you know the difference between PR review and environment approval. PR review is about code quality — is the code correct? Environment approval is about deployment risk — is it safe to deploy this to production? They are separate gates that serve different purposes. A question that says "the agent's PR was approved but the deployment was blocked" is testing this distinction.

5. Least-Privilege Permissions for Agent Capabilities

The principle of least privilege states that every agent should have only the permissions it needs to perform its role — nothing more. This is a core governance concept on the exam.

Where Privilege Is Configured

  • Agent profile (.github/agents/*.agent.md): Each agent's tools list and permissions block define its capabilities
  • Workflow permissions: GitHub Actions permissions: key controls what the GITHUB_TOKEN can do
  • Repository/org-level token permissions: Restrict what tokens agents can use
  • Environment secrets: Sensitive credentials are scoped to specific environments

Applying Least Privilege

Agent Role Needs Does NOT Need
Plan Agent file-read, issue-create, search file-write, shell, environment-select
Dev Agent file-read, file-write, shell, search environment-select, admin
Review Agent file-read, pr-comment, search file-write, shell, deploy
Deploy Agent file-read, shell, environment-select file-write (on source), issue-create
Monitor Agent file-read, web, search file-write, shell, all privileged actions

Common Privilege Mistakes (Exam Traps)

  • Privilege creep: Adding tools to an agent over time without removing unused ones — leads to excessive permissions
  • Copy-paste agent profiles: Reusing an agent profile from another project without auditing tools — gives unnecessary access
  • Overly broad file permissions: Allowing an agent to write to /** instead of a specific directory — widens blast radius
  • Missing tool restrictions: Not configuring restricted commands or allowed directories for the shell tool — agent can run arbitrary commands

📘 Teaching Tip — Audit Agents Regularly: Least privilege is not a one-time configuration. The exam expects you to understand that agent permissions should be audited and reviewed regularly — similar to how you'd review IAM policies for human users. Any time an agent's role changes, its permissions should be re-evaluated. Think of it as permission lifecycle management for agents.

6. Observability, Traceability, and Auditability

Governance isn't just about preventing bad actions — it's also about detecting them and proving what happened. The exam defines three related but distinct concepts:

Observability

The ability to understand the internal state of the agent system from its external outputs. You can observe what agents are doing, how they're performing, and what signals they're producing.

  • Mechanisms: Workflow run logs, agent session logs, status checks, health checks
  • Purpose: Real-time understanding of agent behavior — is the system healthy?

Traceability

The ability to follow the chain of actions from end to end. You can trace a specific change back to the agent that made it, the task that triggered it, and the decision that authorized it.

  • Mechanisms: Commit attribution, PR authorship, workflow run IDs, cross-referenced logs
  • Purpose: Forensic analysis — who did what, when, and why?

Auditability

The ability to prove compliance with policies and standards. You can produce evidence that the system operates within its defined governance boundaries.

  • Mechanisms: GitHub Audit Log, workflow run history, artifact retention, compliance reports
  • Purpose: External verification — can you prove to an auditor that governance was followed?

🎓 Teaching Tip — The Three A's: Remember the distinction: Observability = "Is it working now?" (real-time), Traceability = "What happened?" (forensic), Auditability = "Can we prove it?" (compliance). The exam will use these terms precisely — a question about "proving compliance to an auditor" is about auditability, not observability or traceability.

7. Maintaining Governance Over Time

Governance is not a set-it-and-forget-it activity. The exam expects you to understand how to maintain governance as the agent system evolves.

Governance Lifecycle

  • Initial setup: Define risk profiles, configure guardrails, assign autonomy levels
  • Ongoing monitoring: Review audit logs, analyze failure patterns, check for governance violations
  • Periodic review: Re-evaluate autonomy levels as agents prove their reliability; tighten controls for new risks
  • Incident response: When a governance violation occurs, investigate root cause and update controls

Governance Degradation Patterns

  • Control fatigue: Too many approvals slow down development, leading teams to bypass controls
  • Privilege drift: Agents accumulate permissions over time as new tools are added without removing old ones
  • Ruleset staleness: Rulesets become outdated as the codebase evolves, no longer covering new risk areas
  • Reviewer burnout: CODEOWNERS file becomes stale; reviewers are no longer the right people for the code

Best Practices for Governance Maintenance

  • Schedule regular audits: Quarterly review of agent permissions, rulesets, and autonomy levels
  • Automate governance checks: Use workflows to validate agent profiles against governance policies
  • Document governance decisions: Every autonomy level assignment should have a documented rationale
  • Test guardrails: Periodically attempt to bypass controls to verify they're still working
  • Update CODEOWNERS: Keep file ownership current as team membership changes

⚠️ Exam Trap: A question might describe a scenario where governance has degraded over time and ask: "What's the most likely cause?" The answer is usually privilege drift — agents accumulating permissions without review. The second most common cause is ruleset staleness — governance rules that haven't been updated to cover new capabilities or workflows. Know both patterns.
