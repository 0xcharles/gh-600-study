1. The Five Core Tools + Web & Todo Tools

Every GitHub Copilot agent (whether in the IDE, CLI, or a custom agent) gets access to a set of tools. Tools are how the agent interacts with the outside world — reading files, searching code, editing source, running commands, and even spawning sub-agents. The exam expects you to know the five core tools cold, plus two context-specific ones.

The Five Core Tools (Ranked by Permission Level)

These tools form a permission hierarchy. The rule is simple: always use the least privileged tool that can accomplish the task. This is called least-privilege tool selection, and it's the single most tested concept in Domain 2.

Tool Permission Level What It Does When to Use
read 🟢 Read-only Read files and directories from the workspace You need to see code but not change it
search 🟢 Read-only Search code, issues, PRs, and repository content via semantic or regex matching You need to find something before you can read it
edit 🟡 Write (local files) Create, modify, or delete files in the workspace You need to change code, but don't need to run anything
execute 🟠 Write + Execute Run shell commands, scripts, tests, and build commands in the workspace You need to run code — build, test, lint, install dependencies
agent 🔴 Highest autonomy Delegate tasks to a sub-agent (a new agent loop with its own toolchain) You need to parallelize complex multi-step work or use specialized sub-agents

The Web Tool

The web tool allows the agent to make HTTP requests to external URLs. This is available only when the agent has execute permission or a specific web tool grant in its profile. It's used for fetching documentation, checking APIs, or scraping public web pages. Critical exam note: The web tool is not available by default — it must be explicitly enabled in the agent profile.

The Todo Tool

The todo tool is a lightweight task-management helper that lets the agent create, list, update, and close todo items within a workspace session. It's available only when the agent is operating in a GitHub Workspace or Codespaces environment. Outside those contexts, the todo tool doesn't exist.

🎓 Teaching Tip — The Decision Tree: On the exam, when you see "which tool should the agent use?" walk through this: Can I just look at the data? → read. Do I need to find something first? → search. Do I need to change a file? → edit. Do I need to run a command? → execute. Do I need a sub-agent to manage complexity? → agent. Never choose execute when edit will do — that privilege creep is a classic wrong answer.

⚠️ Exam Alert: The exam will present scenarios where you must choose between execute and edit. Example: "The agent needs to add a new function to auth.py." The correct answer is edit, not execute — you only need to write a file, not run a shell command. If the answer says "use execute to create the file," that's a trap! Execute is for running commands, not editing files.

2. Agent Profile Configuration

Every custom agent on GitHub is defined by an agent profile — a Markdown file with YAML frontmatter stored at .github/agents/*.agent.md. This is how you declare what tools the agent has access to, which MCP servers it can call, which model it uses, and more.

Let me walk you through the anatomy of an agent profile:

--- description: > My code review assistant that checks for security vulnerabilities and suggests fixes. name: security-reviewer tools: - read - search - edit mcp-servers: - vulnerability-db target: github-actions model: gpt-4o disable-model-invocation: false --- # Security Reviewer Agent This agent reviews pull requests for security issues. It checks for OWASP Top 10 patterns and suggests fixes.

Frontmatter Keys — What Each One Does

Key Required? Purpose
description ✅ Required A human-readable description of the agent's purpose. This is what appears in the agent picker UI and is used by the system to match tasks to agents.
name Optional A short identifier for the agent. If omitted, the filename is used (minus .agent.md).
tools Optional List of tools the agent is allowed to use. If omitted, the agent gets the default toolset (read, search, edit). This is the primary way you enforce least privilege.
mcp-servers Optional List of MCP server names the agent can call. These are defined in .mcp.json or .github/mcp.json. If omitted, the agent cannot use any MCP servers.
target Optional Where the agent runs. Common values: github-actions (for CI), codespaces (for dev environments), or cli (for local CLI usage).
model Optional Which model powers this agent (e.g., gpt-4o, claude-sonnet-4). If omitted, the organization's default model is used.
disable-model-invocation Optional Boolean. When set to true, the agent cannot call other models — it's locked to its assigned model. This is a security control to prevent model-hopping.

📘 Teaching Tip — The description field: The exam loves to test that description is the only required field in frontmatter. Everything else is optional. If you see an agent profile missing name, tools, or model, that's fine — but if it's missing description, it's invalid. Remember: description is mandatory, everything else is gravy.

Tools in the Profile — Granular Control

Each tool in the tools list can also be configured with sub-options for even finer-grained access control. For example, you might grant execute but restrict it to only npm test or specific directories. This is the least privilege principle applied at the configuration level.

--- description: Build and test agent for CI pipelines name: ci-builder tools: - read - edit - execute: commands: - npm test - npm run build directories: - src/ - tests/ mcp-servers: [] target: github-actions model: gpt-4o

⚠️ Exam Trap: Some questions will describe an agent profile with tools: [read, search, edit, execute, agent] — ALL five tools — and ask if this is appropriate. Unless the agent needs all five (rare), this is an over-privileged profile. The correct approach is to grant only the tools the agent actually needs.

3. Exam-Critical Artifact Paths (Memorize These)

Now we come to one of the most memorization-heavy parts of the exam. There are a set of file paths in the GitHub ecosystem that serve specific purposes. The exam WILL ask you to match the path to its purpose. Let me break them down into logical groups.

🎓 Instructor's Note: I recommend studying these in groups rather than as a flat list. Notice the pattern — many live under .github/ because that's GitHub's convention directory for repository configuration.

Group 1: Agent & Instructions (Who the Agent Is)

File PathPurpose
.github/copilot-instructions.md Copilot-wide instructions. These are system-level guidelines that apply to ALL Copilot interactions in the repo — IDE, CLI, and agents. Think of this as the "constitution" for how Copilot behaves in this repository.
.github/instructions/*.instructions.md Scoped instructions. Individual instruction files that can be selectively applied to different agents or contexts. More granular than the single copilot-instructions.md.
.github/agents/*.agent.md Agent profile definitions. Each file defines one custom agent — its tools, MCP servers, model, and description. We just covered the anatomy of these files in Section 2.
AGENTS.md Agent registry / index. A top-level file that lists all available agents in the repository. This is the "table of contents" for your agent ecosystem. Agents listed here appear in the GitHub UI agent picker.

Group 2: Prompts & Skills (What the Agent Does)

File PathPurpose
.github/prompts/*.prompt.md Reusable prompt templates. Pre-written prompts that agents or users can invoke. These are the building blocks for agent behavior — think of them as "functions" that an agent can call.
.github/skills/<skill>/SKILL.md Skill definition. A skill is a packaged capability that an agent can learn. Each skill lives in its own subdirectory under .github/skills/ and has a SKILL.md file describing its behavior, inputs, and outputs.

Group 3: Hooks & Workflows (How the Agent Gets Triggered)

File PathPurpose
.github/hooks/*.json Agent lifecycle hooks. JSON files that define behaviors at specific lifecycle events — like pre-agent-run, post-agent-run, on-error. These are the agent equivalent of Git hooks.
.github/workflows/copilot-setup-steps.yml CI setup workflow. A GitHub Actions workflow that runs BEFORE an agent executes in CI. It sets up the environment — installs dependencies, configures tokens, and prepares the workspace.

Group 4: MCP Configuration (How the Agent Connects to Servers)

File PathPurpose
.mcp.json Repository-level MCP configuration. Defines MCP server connections for the entire repository. This is the primary MCP config file.
.github/mcp.json GitHub-scoped MCP configuration. Same format as .mcp.json but lives in the .github/ directory. Used when you want MCP config alongside other GitHub config files.
.vscode/mcp.json VS Code-scoped MCP configuration. MCP server definitions that only apply when the agent runs within VS Code / VS Code Server (including Codespaces). Useful for IDE-specific tooling.

⚠️ Critical Path Memorization: The exam will give you scenarios like "Where do you define reusable prompts for your agent?" The answer is .github/prompts/*.prompt.md. Or "Where do you configure MCP servers?" It could be .mcp.json, .github/mcp.json, or .vscode/mcp.json depending on scope. Know the difference between all of them.

4. MCP Architecture & Servers

MCP stands for Model Context Protocol. It's an open standard that lets AI agents connect to external tools, data sources, and services through a standardized interface. Think of MCP as the USB-C for AI agents — a universal protocol that any agent can use to talk to any service.

MCP Architecture Diagram

Here's the architecture you need to understand for the exam. The agent sends a request through the MCP client, which routes it to the correct MCP server, which in turn calls the external service:

How MCP Servers Work in GitHub

MCP servers in GitHub are configured via MCP configuration files (.mcp.json, .github/mcp.json, or .vscode/mcp.json). Each server entry specifies how to connect to the external service — typically a command (for local servers) or a URL (for remote servers).

{ "mcpServers": { "vulnerability-db": { "type": "stdio", "command": "node", "args": ["mcp-servers/vuln-db-server.js"], "env": { "API_KEY": "${VULN_DB_API_KEY}" } }, "github-remote": { "type": "remote", "url": "https://mcp.github.com/servers/issues" } } }

GitHub Remote MCP Server

GitHub provides its own remote MCP server that exposes GitHub API capabilities as MCP tools. This allows agents to interact with issues, PRs, repositories, and more through the MCP protocol. The remote MCP server is hosted by GitHub and can be referenced by URL in your MCP configuration.

MCP Registries & Allow Lists

Organizations can publish MCP servers to a registry (like the GitHub Marketplace or a private registry) for discoverability. To control which MCP servers agents can use, organizations maintain allow lists — only servers on the allow list can be invoked by agents. This is a critical security control: just like tools, MCP servers follow the principle of least privilege.

📘 Teaching Tip — MCP vs. Direct API: Why use MCP instead of having the agent call the API directly? Three reasons: (1) Standardization — any agent can talk to any MCP server; no custom integration code. (2) Security — MCP servers can enforce authentication, rate-limiting, and audit logging. (3) Discoverability — agents can discover available tools from the MCP server's capability advertisement. The exam will expect you to know these benefits.

5. CI Integration — Copilot CLI in Pipelines

One of the most powerful features of GitHub Copilot agents is the ability to run them in CI/CD pipelines. The Copilot CLI (copilot command) can be invoked inside GitHub Actions workflows, making agents a first-class part of your automated SDLC.

The copilot CLI Command

The command you'll use in CI is:

copilot -p "Review this PR for security vulnerabilities" \ --agent security-reviewer \ --no-ask-user

Key Command-Line Flags

FlagPurpose
-p "<prompt>" The prompt or instruction for the agent. This tells the agent what to do.
--agent <name> Specifies which agent profile to use. The name matches the agent defined in .github/agents/*.agent.md. If omitted, the default agent is used.
--no-ask-user Critical for CI. When running non-interactively (no human to answer questions), this flag tells the agent to proceed autonomously without asking for user input. Without this, the CI job would hang waiting for a response that never comes.

COPILOT_GITHUB_TOKEN

When running in CI, the Copilot CLI needs a GitHub token with appropriate permissions. This is provided via the COPILOT_GITHUB_TOKEN environment variable. Unlike the standard GITHUB_TOKEN (which has limited, job-scoped permissions), the COPILOT_GITHUB_TOKEN should have the permissions the agent needs to do its job — but no more (least privilege again).

name: Agent Review on: [pull_request] jobs: agent-review: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - name: Setup Copilot uses: ./.github/workflows/copilot-setup-steps.yml - name: Run Agent env: COPILOT_GITHUB_TOKEN: ${{ secrets.COPILOT_PAT }} run: | copilot -p "Review this PR for security issues" \ --agent security-reviewer \ --no-ask-user

⚠️ Exam Trap — Token Confusion: The exam will try to confuse GITHUB_TOKEN and COPILOT_GITHUB_TOKEN. Remember: GITHUB_TOKEN is the automatic, short-lived token GitHub creates for each workflow run. It has limited scope. COPILOT_GITHUB_TOKEN is a separate token (usually a PAT or GitHub App token) that you explicitly configure with the permissions the agent needs. If an answer choice says "use the default GITHUB_TOKEN for copilot," it's wrong — you need COPILOT_GITHUB_TOKEN.

6. Execution Contexts

Where and how an agent runs matters enormously for security and governance. The exam tests three execution contexts that you must understand:

Repository Scope

When an agent runs with repository scope, it has access to the entire repository — all files, all branches, all PRs, all issues. This is the default scope for most agents. It's appropriate for agents that need a broad view of the codebase (like code review or refactoring agents).

Branch-Based Scope

With branch-based scope, the agent is restricted to a specific branch. It can read files from that branch, create new branches from it, and push changes to it — but it cannot access other branches. This is ideal for agents that work on feature branches or specific fix branches.

--- description: Feature-specific refactoring agent name: feature-refactor tools: - read - edit target: github-actions # Scope is implicitly branch-based when triggered # by a branch-based event (push, pull_request) ---

CI Workflow Invocation

When an agent is invoked from a CI workflow, it runs in a fresh, ephemeral environment. It has:

  • A clean filesystem (checked-out repository)
  • Temporary credentials (the COPILOT_GITHUB_TOKEN)
  • No access to the developer's local environment
  • Full audit trail (all actions are logged in workflow run logs)

This is the safest execution context because it's fully sandboxed. If the agent does something destructive, it only affects that ephemeral environment.

🎓 Teaching Tip — Context Choice: On the exam, when you're asked which execution context is appropriate for a given task, think: Does this agent need to see the whole codebase or just one branch? Does it run in CI (safe) or in the IDE (connected to a developer's local environment)? CI execution is always preferred for autonomous actions because of its sandboxed, auditable nature.

7. Safe Execution Patterns

Agents can make mistakes. The exam tests your knowledge of how to design agent systems that handle failures gracefully. Here are the patterns you need to know:

Error Handling

Every tool call can fail. Your agent system should have explicit error handling at every level:

  • Tool-level errors: File not found, permission denied, network timeout — the agent should catch these and decide whether to retry or escalate.
  • Application-level errors: Tests fail, build breaks — the agent should stop and report rather than proceeding blindly.
  • System-level errors: Environment crash, resource exhaustion — the CI runner or orchestration layer should handle these.

Retries

Transient failures (network blips, rate limits, temporary service unavailability) should be retried. The standard pattern is exponential backoff with jitter:

# Retry with exponential backoff max_retries = 3 for attempt in range(max_retries): try: result = tool_call(params) break except TransientError: if attempt == max_retries - 1: escalate_to_human(error) break sleep(2 ** attempt + random.uniform(0, 1))

Rollbacks

When an agent makes changes that turn out to be wrong, it should be able to roll back. In the GitHub context, the primary rollback mechanism is:

  • Git revert: If the agent pushed a commit, create a revert commit.
  • Branch deletion: If the agent created a branch with bad changes, delete it.
  • PR close without merge: If the agent opened a PR that hasn't been merged, close it.

Escalation Paths

When an agent cannot resolve an issue (after exhausting retries), it must know who to escalate to. The escalation path is typically:

  1. Retry — try the operation again (up to N times)
  2. Fallback — use an alternative approach or tool
  3. File an issue — create a GitHub Issue with full error details and assign it to the appropriate team
  4. Page the on-call engineer — for critical failures, trigger an alert via the hook system
{ "on-error": { "max-retries": 3, "retry-delay": "exponential", "escalate-after": "retries-exhausted", "escalation-target": { "type": "issue", "assignee": "security-team", "labels": ["agent-escalation", "auto-filed"] } } }

✅ Key Insight — The Exam Pattern: Whenever you see a scenario question involving an agent failure, the correct answer will almost always include an escalation path. Wrong answers will suggest "retry indefinitely" (infinite loops are bad) or "ignore the error" (silent failure is an anti-pattern). Always choose the answer that includes logging + retry (bounded) + escalation.

8. Autonomous Actions — Creating Branches & PRs

The most autonomous thing a GitHub agent can do is create branches and pull requests without human intervention. This is where the tool permissions, agent profile, and execution context all come together.

Creating a Branch

An agent with execute permission can create branches via git commands in the workspace. But more commonly, the agent uses the GitHub remote MCP server or direct API calls to create branches through the GitHub API:

# Agent creates a fix branch via execute tool git checkout -b fix/auth-timeout git commit -m "Fix auth timeout bug" git push origin fix/auth-timeout

Creating a Pull Request

After the agent has made changes on a branch, it can open a PR. The agent can use the gh CLI (via execute tool) or the GitHub API:

# Agent opens a PR via gh CLI gh pr create \ --title "Fix auth timeout bug" \ --body "Automated fix by security-reviewer agent" \ --base main \ --head fix/auth-timeout \ --label "auto-pr" \ --reviewer security-team

Autonomy vs. Safety in Autonomous Actions

Creating branches and PRs autonomously is powerful but requires guardrails:

  • Branch naming conventions: Enforce patterns like agent/<agent-name>/<description> so agent-created branches are easily identifiable.
  • PR templates: Require agents to use PR templates that include the agent name, tool list, and a summary of changes.
  • CODEOWNERS: Set automatic reviewers for agent-created PRs.
  • Rulesets: Apply branch protection rules that prevent agents from pushing directly to main — agents must always go through PRs.
# Branch protection ruleset (configured in GitHub UI or as code) # Agents can push to agent/* branches but NOT to main - pattern: agent/* allows_force_pushes: false requires_pr: false - pattern: main requires_pr: true required_reviewers: ["security-team"] dismisses_stale_reviews: true

⚠️ Exam Critical — Autonomy Levels: The exam maps autonomy directly to tool permissions: read/search = no autonomy (advisory only), edit = limited autonomy (can change files but not execute), execute = moderate autonomy (can run commands and make changes), agent = full autonomy (can delegate to sub-agents). Creating branches and PRs requires at minimum execute permission. If an exam question asks what tool an agent needs to open a PR, the answer is execute (to run gh pr create).

9. Putting It All Together — Scenario Walkthrough

Let's walk through a real exam-style scenario to see how all these concepts connect.

🎓 Scenario: Your team wants a "dependency-updater" agent that scans for outdated dependencies, updates them, runs tests, and creates a PR. How do you configure this agent?

Step 1 — Create the agent profile at .github/agents/dependency-updater.agent.md:

--- description: > Scans for outdated dependencies, updates them to latest compatible versions, runs the test suite, and creates a PR. name: dependency-updater tools: - read - search - edit - execute: # Needs execute to run npm audit, npm test, and gh pr create commands: - npm audit - npm update - npm test - gh pr create mcp-servers: - npm-registry # To query npm registry for latest versions target: github-actions model: gpt-4o

Step 2 — Configure MCP server in .github/mcp.json:

{ "mcpServers": { "npm-registry": { "type": "remote", "url": "https://mcp.npmjs.com/registry" } } }

Step 3 — Create a CI workflow that runs the agent on a schedule:

name: Dependency Update on: schedule: - cron: '0 6 * * 1' # Every Monday at 6 AM UTC workflow_dispatch: # Manual trigger too jobs: update: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-node@v4 - run: npm install - name: Run Dependency Update Agent env: COPILOT_GITHUB_TOKEN: ${{ secrets.COPILOT_PAT }} run: | copilot -p "Check for outdated dependencies, \ update them, run tests, and create a PR" \ --agent dependency-updater \ --no-ask-user

Step 4 — Add guardrails:

  • Add .github/PULL_REQUEST_TEMPLATE/agent-pr.md so agent-created PRs always include a summary
  • Set up CODEOWNERS so agent PRs are reviewed by the platform team
  • Use rulesets to prevent the agent from pushing directly to main

Step 5 — Error handling: Configure a hook at .github/hooks/on-error.json to file an issue if the agent fails:

{ "on-error": { "max-retries": 2, "escalate-after": "retries-exhausted", "escalation-target": { "type": "issue", "repository": "my-org/my-repo", "assignee": "platform-team", "labels": ["dependency-update", "agent-failure"] } } }

✅ Takeaway: This scenario touches every concept in Domain 2 — tool selection, agent profiles, MCP servers, CI integration, execution context, safe patterns, and autonomous actions. If you can walk through this scenario confidently, you're well-prepared for the exam questions.
