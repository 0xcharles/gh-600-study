Agent Architecture &
SDLC Processes

Learn how to design agent workflows that integrate safely into software delivery — defining planning vs execution boundaries, configuring governance, and building reliable, observable agent systems that your team can trust.

1. Agent Responsibilities in the SDLC

Let's start with the big picture. An agentic AI system is not a chatbot that answers questions — it's an autonomous system that takes actions against real infrastructure. In the SDLC context, agents participate in every stage of software delivery: planning, coding, building, testing, releasing, deploying, operating, and monitoring.

The exam wants you to understand which stages are good for agents and which ones need human oversight. Here's the breakdown:

Stages Where Agents Excel

  • Planning & Triage: Agents can scan issue queues, classify bugs, suggest labels, and auto-assign work based on skill matching. They can also research relevant code before a human picks up a ticket.
  • Coding (Controlled): Agents generate code, write tests, and refactor — but always inside a branch with guardrails. They never commit directly to main.
  • Building & Testing: Agents can trigger CI workflows, parse build logs, detect regressions, and auto-retry flaky tests.
  • Release & Deploy: With the right autonomy level, agents can cut releases, promote artifacts between environments, and roll back on failure.
  • Monitoring: Agents watch dashboards, correlate alerts, and file new issues when anomaly patterns emerge.

Stages Where Agents Are Less Suitable

  • Strategic Planning: Agents don't decide the product roadmap. That requires human context, business judgment, and stakeholder alignment.
  • Code Review: Agents can assist with review (linting, security scanning, diff analysis), but the final sign-off must be human.
  • Incident Response (Severe): While agents can triage, the actual decision to rollback, hotfix, or page an on-call engineer requires human judgment.
  • Production Deployments (Unsupervised): Pushing to production without any human gate is almost always an anti-pattern (unless you have extremely mature rollback automation).

📘 Teaching Tip: The exam will give you a scenario and ask: "Is this a good task for an agent?" The decision rule is: Does the task have a clear input, a defined process, and a checkable output? If yes, an agent can probably do it. If the success criteria are fuzzy or require human empathy, keep the human in the loop.

2. Anti-Patterns to Recognize

Domain 1 specifically tests your ability to identify and avoid common agent anti-patterns. These show up in case studies where something goes wrong, and you need to choose the right fix.

🔄 Autonomy Mismatch

Giving an agent more autonomy than a task requires — or less than it needs to be useful. For example, giving a code-generation agent full shell access when it only needs file-write permissions. The result: unnecessary risk. On the flip side, giving a monitoring agent read-only access when it needs to remediate means it can alert but never fix.

Fix: Match the minimum autonomy level to the task's actual requirements. We'll cover autonomy levels in depth later.

📋 Plan-as-Checklist

An agent generates a plan, the human approves it, and then the agent executes exactly what it said — even if conditions change during execution. This treats the plan as a rigid checklist instead of a set of intended outcomes.

Fix: The plan is a reviewable intent, not a fixed contract. Agents should re-evaluate against current state and flag deviations.

📈 Scope Creep

The agent starts with one task ("fix the bug in auth"), then widens into refactoring the entire auth module, adding tests, changing the database schema, and deploying a new version — all in one session without checkpointing or human approval.

Fix: Break large tasks into small, discrete steps. Use checkpoints. Require human approval before crossing scope boundaries.

🔇 Silent Failure

An agent encounters an error and quietly retries, gives up, or produces a degraded result without notifying anyone. This is dangerous because downstream processes depend on the agent's output.

Fix: Always log failures, surface them to the orchestration layer, and escalate to a human if retries are exhausted. Never let an agent silently fail.

⚠️ Exam Warning: On the case study sections, you'll almost certainly see a scenario involving silent failure or autonomy mismatch. The wrong answer will sound reasonable but skip the governance step. Always pick the answer that includes observability, logging, or human escalation.

3. Input, Output & Success Criteria

Every agent task must have well-defined inputs, expected outputs, and measurable success criteria. If any of these three is vague, the agent will produce unpredictable results. Here's a reference table:

Task Input Expected Output Success Criteria
Classify incoming issues Issue title + body Label assignment, priority score Accuracy > 90% vs human baseline
Generate unit tests Source file, function signatures Test file with assertions Tests pass, coverage gain > 5%
Write PR description Diff, commit messages Summary, change log, testing notes Follows team template, human approves
Review code for security PR diff, dependency manifest List of vulnerabilities found Detected > 80% of known vulns in test set
Cut release + deploy to staging Release tag, environment config Deployment URL, health check pass All smoke tests green, no rollback

Notice the pattern: success criteria are quantitative and verifiable. Not "make the code better" — that's not measurable. Always define success in terms a CI pipeline or a reviewer can check.

4. Planning vs Execution Separation

This is one of the most important concepts in Domain 1 — and it appears in multiple exam questions. The core principle is:

✅ Core Principle: A plan is a reviewable intent — it describes what the agent proposes to do. Execution is the act of changing state — writing files, running commands, making API calls. These two phases must be separated by a human review gate.

Why does this matter? Because a plan costs nothing to produce, but execution has real-world consequences. A plan that deletes a file is safe to review; actual deletion is destructive. By separating the phases, you get:

  • Safety: Humans can catch mistakes before they happen.
  • Auditability: You have a record of what was proposed vs what was actually done.
  • Collaboration: Multiple humans can weigh in on the plan before execution starts.

How It Works in Practice

A typical separated workflow looks like this in a GitHub Actions context:

name: Agent Planning Session on: workflow_dispatch: inputs: task_description: description: 'Describe the task for the agent' required: true jobs: plan: runs-on: ubuntu-latest steps: - name: Generate plan from task run: | echo "## Agent Plan" > plan.md echo "### Proposed Changes" >> plan.md echo "1. Create src/utils/validator.py" >> plan.md echo "2. Add unit tests in tests/test_validator.py" >> plan.md echo "3. Update requirements.txt" >> plan.md echo "" >> plan.md echo "### Risk Assessment" >> plan.md echo "- No breaking changes expected" >> plan.md echo "- All new code covered by tests" >> plan.md - name: Upload plan as artifact uses: actions/upload-artifact@v4 with: name: agent-plan path: plan.md # Human reviews the plan artifact before approving execution

After the human approves, a separate execution workflow runs, using needs to depend on the plan workflow's approval:

name: Agent Execution on: workflow_run: workflows: ["Agent Planning Session"] types: [completed] # Only runs if the plan workflow completed successfully jobs: execute: if: ${{ github.event.workflow_run.conclusion == 'success' }} runs-on: ubuntu-latest steps: - name: Check out repository uses: actions/checkout@v4 - name: Execute agent tasks run: | # Agent generates code, writes files, runs tests python3 -c " # Agent creates the validator module import os os.makedirs('src/utils', exist_ok=True) with open('src/utils/validator.py', 'w') as f: f.write('# Validator module\n') print('Execution complete') " - name: Create Pull Request uses: peter-evans/create-pull-request@v6 with: branch: agent-execution-${{ github.run_id }} title: 'Agent: Implement validator module' body-path: plan.md

🧠 Teaching Tip: Remember the exam slogan: "Plan is reviewable intent; execution changes state." If you see an exam answer that combines planning and execution into a single step without a human gate, that's the wrong answer — even if it's faster or more efficient.

5. Autonomy Levels

Not all agents need the same level of freedom. The exam defines three autonomy levels that map to the capabilities an agent is granted:

Level Allowed Operations Example Tools When to Use
Low Read, search, summarize File reader, grep, GitHub issue viewer, code search Triage, research, documentation queries
Medium Low + edit, execute approved scripts, create PRs File writer, command runner, PR creator, test runner Code generation, bug fixes, refactoring with review
High Medium + agent delegation, MCP servers, shell commands Sub-agent spawner, MCP tools, unrestricted shell, deployment triggers Complex workflows, multi-step automation, autonomous deployments (with guardrails)

Choosing the Right Level

The rule of thumb: start low, escalate only when needed. If a task can be accomplished with read-only access, don't give the agent write permissions. If a task needs to create a PR but doesn't need shell access, use Medium — not High.

Here's how you might configure autonomy levels in an agent's YAML configuration:

# .github/agent-config.yaml agent: name: "triage-agent" autonomy_level: "low" # Options: low | medium | high # Low-level capabilities tools: - type: file_reader allowed_paths: ["src/**", "tests/**"] - type: search max_results: 50 # Medium capabilities (only if autonomy_level >= medium) elevated_tools: - type: file_writer allowed_paths: ["src/**", "tests/**"] max_file_size: 1MB - type: pr_creator require_reviewers: true require_checks: true # High capabilities (only if autonomy_level == high) full_tools: - type: shell allowed_commands: ["npm test", "python -m pytest", "git status"] deny_list: ["rm -rf", "sudo", "> /dev/sda"]

⚠️ Exam Trap: The exam will test whether you can match autonomy level to a scenario. If the scenario says "an agent that researches a bug" — that's Low. If it says "an agent that writes a fix and opens a PR" — that's Medium. If it says "an agent that edits infrastructure and deploys" — that's High. Don't over- or under-assign.

6. Task Quality: Weak vs Well-Defined

An agent is only as good as the task you give it. Vague instructions produce unpredictable outcomes. The exam expects you to distinguish between weak tasks and well-defined tasks.

Weak Tasks (Avoid These)

  • "Fix the login page" — What's wrong with it? What does "fixed" mean?
  • "Improve performance" — By how much? Measured how? In what scope?
  • "Review the code" — What kind of review? Security, style, correctness?
  • "Make it better" — This isn't a task at all.

Well-Defined Tasks (Good Examples)

  • "Add input validation to src/auth/login.ts that rejects email addresses longer than 254 characters and returns a 400 error."
  • "Generate unit tests for src/services/payment.ts that achieve at least 80% branch coverage. Tests must pass on CI."
  • "Scan requirements.txt for packages with known CVSS 7+ vulnerabilities and open an issue for each one with remediation guidance."

The Checklist for Well-Defined Tasks

When writing a task for an agent, check that it includes:

  1. Scope: Which files or services are in scope?
  2. Action: What exactly should the agent do?
  3. Constraints: What must the agent NOT do?
  4. Quality bar: How do we know it's done well?
  5. Output format: What does the deliverable look like?
# ❌ Weak task Task: Fix the error handling in the API # ✅ Well-defined task Task: Improve error handling in src/routes/users.ts Scope: - File: src/routes/users.ts - Do not modify any other files Action: - Wrap each route handler in a try/catch block - Return consistent JSON error responses: { "error": string, "code": integer } - Log each error with context using the existing logger Constraints: - Do not change the route signatures or response shapes for success cases - Do not add external dependencies Quality bar: - All existing tests must still pass - New error responses must match the format in /docs/api-errors.md - No lint warnings introduced

7. PR-Based Governance

Pull Requests are the primary governance mechanism for agent actions. Every change an agent makes should flow through a PR — never a direct push. This gives you a structured system of checks and balances.

PR Templates

Templates ensure agents produce consistent, reviewable PRs. An agent-generated PR should follow the same template as a human PR, describing what changed and why:

# .github/PULL_REQUEST_TEMPLATE.md ## Summary <!-- Agent: Summarize the changes in 2-3 sentences --> ## Changes - [ ] Feature - [ ] Bug fix - [ ] Refactor - [ ] Tests ## Files Modified <!-- Agent: List changed files --> ## Testing - [ ] Unit tests added/passed - [ ] Integration tests passed - [ ] Manual testing performed ## Agent Metadata - **Agent ID:** {{ agent_id }} - **Task ID:** {{ task_id }} - **Autonomy Level:** {{ autonomy_level }}

Required Status Checks

Every agent PR should require passing checks before merge. This includes CI jobs, linting, security scanning, and test coverage thresholds:

# Branch protection rule (configured in repo settings or rulesets) - Require status checks before merging: ✅ build-and-test (ubuntu-latest) ✅ lint-and-format ✅ security-scan (codeql) ✅ coverage-check (>= 80%) - Require at least 1 approval from CODEOWNERS - Block merge if any check fails

CODEOWNERS

CODEOWNERS files define who (or which team) must approve changes to specific files. This is critical when an agent touches sensitive code paths:

# .github/CODEOWNERS # Global owners for review * @core-team # Security-critical paths require security team approval src/auth/** @security-team src/secrets/** @security-team # Deployment configs require platform team deploy/** @platform-team infra/** @platform-team # Agent-specific code is owned by the AI-Infra team .github/agents/** @ai-infra-team

Rulesets and Environment Gates

Repository rulesets allow you to enforce fine-grained policies — like requiring a 6-hour cool-down period before merging agent PRs, or requiring approval from two different teams. Environment protection rules gate deployments to production behind manual approvals, ensuring an agent can't promote a build without a human saying "go."

🔑 Exam Tip: You'll be asked to choose the right governance mechanism for a scenario. If the scenario says "prevent direct pushes to main" — that's a branch protection rule. If it says "require specific reviewers for security files" — that's CODEOWNERS. If it says "require manual approval before production deploy" — that's an environment gate. Know the difference.

8. Reliable Workflows

Reliable agent workflows are built on four pillars: outputs, contexts, triggers, and cross-job handoffs. Let's break each one down.

Outputs

Every job in a workflow can produce outputs that downstream jobs consume. This is how you pass data like plan summaries, test results, or artifact paths between jobs:

jobs: agent-plan: runs-on: ubuntu-latest outputs: plan_summary: ${{ steps.generate.outputs.summary }} branch_name: ${{ steps.create_branch.outputs.branch }} steps: - id: generate run: echo "summary=Fix null-pointer in UserService.process()" >> $GITHUB_OUTPUTS - id: create_branch run: echo "branch=fix-null-ptr-user-service" >> $GITHUB_OUTPUTS agent-execute: needs: [agent-plan] runs-on: ubuntu-latest steps: - run: | echo "Executing plan: ${{ needs.agent-plan.outputs.plan_summary }}" echo "On branch: ${{ needs.agent-plan.outputs.branch_name }}"

Contexts

GitHub Actions provides rich contextual information — github (event payload, repo, actor), env (environment variables), vars (organization/repo variables), and secrets (sensitive values). Agents use these contexts to make decisions:

steps: - name: Agent reads context run: | echo "Event: ${{ github.event_name }}" echo "Branch: ${{ github.ref_name }}" echo "Actor: ${{ github.actor }}" echo "SHA: ${{ github.sha }}" echo "Repo: ${{ github.repository }}" # Use contexts in conditional logic if [ "${{ github.actor }}" == "github-actions[bot]" ]; then echo "This is an automated agent run" fi

Triggers

Workflows can be triggered by events (push, pull_request, issues, schedule), workflow_dispatch (manual), workflow_run (chained), or repository_dispatch (external webhook). Each trigger type serves a different purpose for agent workflows:

# Trigger on issue creation (planning phase) on: issues: types: [opened, labeled] # Trigger on PR from an agent branch on: pull_request: branches: [agent-*] # Chain workflows: execution runs after plan approval on: workflow_run: workflows: ["Agent Plan"] types: [completed] # External trigger (e.g., from an agent orchestrator) on: repository_dispatch: types: [agent-execute-task]

Cross-Job Handoffs with needs

The needs keyword creates a DAG (directed acyclic graph) of job dependencies. A downstream job only runs after its upstream dependencies succeed. This is essential for multi-step agent flows:

jobs: validate-input: runs-on: ubuntu-latest outputs: valid: ${{ steps.check.outputs.is_valid }} steps: - id: check run: echo "is_valid=true" >> $GITHUB_OUTPUTS generate-code: needs: [validate-input] runs-on: ubuntu-latest steps: - run: echo "Generating code..." run-tests: needs: [generate-code] runs-on: ubuntu-latest steps: - run: echo "Running tests..." create-pr: needs: [run-tests] runs-on: ubuntu-latest steps: - run: echo "Creating PR..." # If run-tests fails, create-pr never executes notify: if: always() needs: [create-pr] runs-on: ubuntu-latest steps: - run: echo "Pipeline completed. Status: ${{ job.status }}"

Notice the notify job uses if: always() — this is a reliability pattern that ensures notification even if upstream jobs fail. We'll cover more reliability patterns next.

9. Agent Operations: Observability, Governance & Reliability

The final section of Domain 1 covers the operational side — how you run agents reliably in production. This includes observability, tool governance, secrets management, hooks, and reliability patterns.

Observability

You can't manage what you can't see. Every agent action should produce structured logs that feed into your observability stack. Key metrics to track:

  • Execution time per task — detect slowdowns
  • Success/failure rate — spot regressions
  • Tool call frequency — identify abuse patterns
  • Human intervention rate — measure autonomy effectiveness
  • Token usage / API costs — track resource consumption
# Structured logging example { "timestamp": "2026-06-01T14:30:00Z", "agent_id": "fixer-agent-v2", "session_id": "sess_abc123", "event": "tool_call", "tool": "file_writer", "target": "src/auth/login.ts", "duration_ms": 342, "success": true, "tokens_used": 1250, "cost_usd": 0.025 }

Tool Governance

Not all tools should be available to all agents. Tool governance means defining allow lists and deny lists at the agent level, and auditing tool usage. A common approach:

  • Allow lists: Only approved tools can be called (e.g., file_reader, file_writer, command_runner).
  • Deny lists: Explicitly dangerous tools are blocked (e.g., sudo, rm -rf, network scanners).
  • Rate limits: Cap the number of tool calls per minute to prevent runaway agents.

Secrets Management

Agents should never have direct access to secrets. Instead, use GitHub Actions secrets and pass them only when needed. Never log or expose secrets in output:

steps: - name: Agent uses a secret via environment variable env: # Secret is injected by GitHub, not by the agent DEPLOY_KEY: ${{ secrets.DEPLOY_KEY }} run: | # Agent can use DEPLOY_KEY but cannot read it echo "Deploying with key length: ${#DEPLOY_KEY}" # ❌ Never do this: echo "Key is $DEPLOY_KEY" # ✅ Safe: use the key via the tool that accepts it - name: Install SSH key securely uses: webfactory/ssh-agent@v0.9.0 with: ssh-private-key: ${{ secrets.SSH_PRIVATE_KEY }}

Hooks

Hooks allow you to inject custom logic at key points in the agent lifecycle — before a tool runs, after a tool completes, or when an error occurs. This is how you enforce governance policies:

# Hook configuration in agent config hooks: pre_tool: - name: validate_file_path script: | if [[ "$TOOL_TARGET" != src/* && "$TOOL_TARGET" != tests/* ]]; then echo "ERROR: Agent attempted to modify unauthorized path: $TOOL_TARGET" exit 1 fi post_tool: - name: log_usage script: | echo "Agent $AGENT_ID called $TOOL_NAME on $TOOL_TARGET in $(duration)s" on_error: - name: notify_team script: | curl -X POST -H "Content-Type: application/json" \ -d '{"text":"Agent $AGENT_ID failed: $ERROR_MESSAGE"}' \ ${{ secrets.SLACK_WEBHOOK }}

Reliability Patterns

Production agent systems need reliability engineering. Here are the key patterns:

Pattern Description Implementation
Retry with Backoff Automatically retry failed operations with increasing delay Exponential backoff: 1s, 2s, 4s, 8s — max 5 retries
Circuit Breaker Stop calling a failing service after N consecutive failures After 10 failures, open circuit for 60s, then half-open test
Dead Letter Queue Route failed tasks to a separate queue for human review Create an issue with failure details and assign to on-call
Timeout Enforcement Prevent agents from running indefinitely Set max execution time per step (e.g., 5 minutes)
Checkpointing Save agent state at regular intervals for resume capability Persist session state to GitHub Artifacts or external store

✅ Key Insight: The exam considers agent operations as a systems engineering problem, not an AI problem. The same reliability patterns you'd apply to a microservice (retries, circuit breakers, timeouts) apply to agents. If an answer choice describes an agent architecture without mentioning observability or error handling, it's incomplete.
