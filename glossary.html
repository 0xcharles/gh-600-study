GH-600 Glossary of Terms

Every exam-critical term defined in context. Each entry includes exam context, related file paths, and domain tags. Use the search box to quickly find any term.

A Agent Domain 1
An autonomous AI system that takes actions against real infrastructure within the SDLC. Unlike chatbots that only answer questions, agents execute tool calls — reading, editing, running commands, delegating sub-tasks, and persisting results. The exam defines agents by their autonomy level and tool set.
A Agent Profile Domain 2
A Markdown file with YAML frontmatter at .github/agents/*.agent.md that defines an agent's description (required), name, tools, mcp-servers, target, model, and disable-model-invocation. The description field is the only required field.
.github/agents/*.agent.md
A Autonomy Level Domain 1
Three-tier classification of agent freedom: Low (read, search only), Medium (adds edit, execute approved scripts, create PRs), High (adds sub-agent delegation, MCP servers, unrestricted shell). The rule: start low, escalate only when needed.
A Autonomy Mismatch Domain 1
An anti-pattern where an agent is granted more autonomy than a task requires (over-privileged) or less than it needs (under-privileged). Over-privileged agents create unnecessary risk; under-privileged agents can't complete their work. Fix: match the minimum autonomy level to the task's actual requirements.
B Branch Protection Domain 6
GitHub rules that prevent direct pushes to protected branches (typically main or production). Requirements include: pull request reviews, passing status checks, up-to-date branches, and no force pushes. Essential for agent governance — agents should never bypass branch protection.
B Bypass List Domain 6
A configured list of entities (users, teams, apps) that are exempt from specific ruleset or branch protection requirements. In agent contexts, a bypass list might allow a specific CI agent to merge without human review while requiring it for all other agents. Use sparingly — bypass lists reduce governance.
C Case Study Domain 1
A scenario-based exam question format on the GH-600. Each case study presents a realistic agent system scenario with multiple sub-questions. You must analyze the situation, identify problems, and choose the correct architectural decisions. There are 2 case studies on the exam (56 questions total).
C CI Integration Domain 2
The process of running agents within GitHub Actions CI pipelines. Configured via .github/workflows/copilot-setup-steps.yml which sets up the environment before agent execution. CI agents use token-based authentication (GitHub App installation tokens or PATs) and operate with short-term memory only (no cross-run persistence).
.github/workflows/copilot-setup-steps.yml
C Concurrency Domain 6
The ability of multiple agents to operate simultaneously on the same repository or workflow. Concurrency introduces risks of conflict (two agents editing the same file), race conditions, and coordination failures. Managed through isolation strategies, branch strategies, and GitHub Actions concurrency groups.
C Context Drift Domain 3
When an agent's internal model of the world diverges from reality. Caused by stale memory, external state changes, concurrent modifications, or erroneous assumptions. Detection strategies include re-read confirmation, timestamp checking, and consistency validation. Correction: re-read → re-validate → re-plan.
C Control Plane Domain 1
The central management layer that governs all agent activity. GitHub serves as the control plane for agentic AI systems — it manages agent registration, authentication, authorization, audit logging, and governance policies. The control plane is distinct from the data plane (where agents execute actions).
C Copilot Memory Domain 3
GitHub's long-term memory implementation for Copilot agents. Stores repository-level facts (key files, architecture patterns, API contracts) and user-level preferences (coding style, testing conventions). Has a 28-day auto-deletion policy — memories not reinforced within 28 days are automatically pruned.
D Durable Artifact Domain 3
A permanent record of agent activity that survives session termination. Includes commits (Git history), PRs (change proposals with reviews), workflow logs (CI execution records, 90-day retention), and issues/comments (persistent threads). Durable artifacts are the system of record for agent actions.
E Environment Protection Domain 6
GitHub deployment protection rules that gate deployments to specific environments (e.g., production, staging). Rules include: required reviewers, waiting periods, and deployment branch restrictions. Critical for agent governance — prevents agents from deploying without human approval.
E Error Handling Domain 2
The retry, rollback, and escalation patterns that govern what an agent does when a tool call fails. Best practices: implement exponential backoff for transient errors, rollback changes on critical failures, and escalate to a human when retries are exhausted. Silent failure (agent fails without notification) is a key anti-pattern.
E Evaluation Domain 4
The process of measuring agent output against defined success criteria. Core to the Plan-Act-Evaluate (PAE) loop. Uses evaluation signals (functional, quality, compliance, process) to determine if an agent performed correctly. Includes automated scanning (CodeQL, Dependabot, Secret Scanning) and human review (PR review).
E execute (Tool) Domain 2
The tool that allows the agent to run shell commands, scripts, tests, and build commands in the workspace. Permission level: Write + Execute (orange). Used when the agent needs to run code. Not for editing files — that's the edit tool. Exam tip: Never choose execute when edit will do — that's privilege creep.
E edit (Tool) Domain 2
The tool that allows the agent to create, modify, or delete files in the workspace. Permission level: Write (yellow). Used when the agent needs to change code but doesn't need to run anything. Part of the five core agent tools in the least-privilege hierarchy.
A agent (Tool) Domain 2
The tool that allows an agent to delegate tasks to a sub-agent — spawning a new agent loop with its own toolchain. Permission level: Highest autonomy (red). Used to parallelize complex multi-step work or use specialized sub-agents. Highest-risk tool — requires strong governance.
R read (Tool) Domain 2
The tool that allows the agent to read files and directories from the workspace. Permission level: Read-only (green). Used when the agent needs to see code but not change it. The lowest-privilege tool in the hierarchy — always prefer read when the task only requires information gathering.
S search (Tool) Domain 2
The tool that allows the agent to search code, issues, PRs, and repository content via semantic or regex matching. Permission level: Read-only (green). Used when the agent needs to find something before it can read it. Unlike read which requires a known path, search discovers content across the repository.
F Fleet (/fleet) Domain 5
A group of agents managed as a single unit within a GitHub organization. Fleet management allows centralized configuration, monitoring, and policy enforcement across multiple agents. The /fleet endpoint or concept relates to multi-agent orchestration and fleet-wide coordination.
/fleet (endpoint / management concept)
G Guardrails Domain 6
The safety boundaries and governance controls that constrain agent behavior. Include: rulesets (repo-level policies), branch protection (prevent direct pushes), environment protection (deployment gates), CODEOWNERS (required reviewers), and autonomy levels (capability limits). Guardrails are the primary mechanism for safe agent deployment.
L Least Privilege Domain 2
The security principle that an agent should be granted only the minimum permissions necessary to complete its task. Applied through tool selection (choose read over edit, edit over execute), tool configuration (restrict commands and directories), and autonomy levels. The single most tested concept in Domain 2.
L Long-Term Memory Domain 3
Tier 2 of the three-tier memory hierarchy. Persists across sessions via Copilot Memory. Stores repository-level facts and user-level preferences. Has a 28-day TTL with auto-deletion. Distinguished from short-term memory (session-scoped, volatile) and durable artifacts (permanent records).
M Matrix Strategy Domain 2
A GitHub Actions feature that allows running a job across multiple combinations of variables (e.g., different OS, runtime versions, environments). In agent contexts, a matrix strategy enables testing agent behavior across multiple configurations simultaneously, providing comprehensive evaluation signals in a single workflow run.
M MCP Domain 2
Model Context Protocol — an open standard that lets AI agents connect to external tools, data sources, and services through a standardized interface. Described as "USB-C for AI agents". MCP architecture: Agent → MCP Client → MCP Server → External Service. Configured via .mcp.json, .github/mcp.json, or .vscode/mcp.json.
.mcp.json · .github/mcp.json · .vscode/mcp.json
M MCP Allow List Domain 2
A security control that specifies which MCP servers an agent is permitted to use. Defined in the agent profile under the mcp-servers field. An empty list means no MCP access. This is the primary mechanism for least privilege in MCP context — agents should only connect to servers they actually need.
M MCP Registry Domain 2
A centralized catalog of available MCP servers that agents can discover and connect to. The registry provides server metadata, connection details, authentication requirements, and capability descriptions. Enables dynamic server discovery — agents can find and connect to registered servers without hard-coded configuration.
M MCP Server Domain 2
A service that implements the Model Context Protocol, exposing tools, data, or capabilities that agents can consume. Each MCP server has a name, host, port, authentication method, and set of available tools. Defined in MCP configuration files (.mcp.json) and referenced in agent profiles via the mcp-servers field.
N needs Domain 2
A GitHub Actions keyword that defines job dependencies. In agent workflows, needs is used to separate planning from execution — a plan job runs first, the human reviews, then the execution job runs if: needs.plan.result == 'success'. This enforces the plan-review-execution gate pattern.
P Plan-Act-Evaluate (PAE) Domain 4
The core feedback loop for all agentic AI systems. Plan: Agent generates a proposed approach. Act: Agent executes the plan (with human review gate). Evaluate: Agent's output is measured against success criteria. The loop repeats iteratively to improve agent quality. The foundational model for Domain 4.
P Plan-as-Checklist Domain 1
An anti-pattern where an agent treats its plan as a rigid checklist rather than a set of intended outcomes. The agent generates a plan, the human approves it, and then the agent executes exactly what it said — even if conditions change. Fix: treat plans as reviewable intent, not fixed contracts. Agents should re-evaluate against current state.
R Root Cause Analysis Domain 4
The process of tracing agent failures from symptom → root cause → fix. Uses a structured decision framework: Is the configuration correct? Are the instructions complete? Are evaluation signals catching the right things? The exam tests your ability to match symptoms (e.g., "agent produces wrong code") to root causes ("instructions too vague") and fixes ("add detailed instructions").
R Ruleset Domain 6
A configurable set of policies enforced at the repository or organization level. Rulesets control: branch protection, required checks, merge requirements, commit signing, and file-level restrictions. More flexible than traditional branch protection — rulesets can target specific users, teams, or apps including bypass lists for agent contexts.
S Session Logs Domain 4
The complete timestamped record of an agent's interaction — every prompt, every tool call, every output, every error. Used for post-mortem analysis, debugging failures, and auditing agent decisions. The first diagnostic step when an agent produces unexpected output is always: check the session logs.
S Short-Term Memory Domain 3
Tier 1 of the three-tier memory hierarchy. The agent's working memory for the current conversation or session. Includes conversation history, in-session decisions, intermediate results, and temporary state. Volatile — disappears when the session ends. Also called session memory. Do not confuse with Copilot Memory (long-term).
S System of Record Domain 1
The authoritative data source for agent activity. GitHub serves as the system of record — all agent actions (commits, PRs, issues, workflow runs, comments) are recorded permanently. The control plane governs the system of record. Durable artifacts are the evidence stored in the system of record.
T Tool Domain 2
A capability that an agent can use to interact with the outside world. The five core tools are read, search, edit, execute, and agent. Additional tools include web (HTTP requests, must be explicitly enabled) and todo (task management, workspace-specific). Tools form a permission hierarchy from read-only (lowest) to agent (highest).
T Tool Tuning Domain 4
One of the four tuning levers for improving agent performance. Involves adjusting which tools an agent has access to, adding tool-specific constraints (restricted commands, allowed directories), enabling/disabling the web tool, or adding/removing the agent tool. Second priority lever after instructions — lower risk than environment or MCP changes.
