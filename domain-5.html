Multi-Agent Systems
& Orchestration

One agent is a tool. Multiple agents are a system. Domain 5 teaches you how to design, coordinate, and troubleshoot multi-agent workflows — deciding which agent does what and when, while managing isolation, conflicts, and observability across the entire pipeline.

1. Multi-Agent Responsibilities in the SDLC (Software Development Lifecycle)

In a modern agentic SDLC (Software Development Lifecycle), different agents specialize in different stages of the software lifecycle. Rather than one monolithic agent doing everything, you build a team of agents — each with its own profile, tools, permissions, and responsibilities. The exam expects you to understand how these roles map to SDLC phases.

Common Agent Roles Across the SDLC

SDLC Phase Agent Role Typical Tools
Planning Plan Agent — analyzes requirements, breaks down work, creates issues and milestones file-read, issue-create, web
Development Dev Agent — writes code, runs tests, commits changes file-read, file-write, shell, search
Review Review Agent — inspects PR diffs, runs linters, flags issues pr-comment, search, web
Testing Test Agent — writes and runs unit, integration, and E2E tests shell, file-write, search
Deployment Deploy Agent — builds artifacts, deploys to environments shell, web, environment-select
Monitoring Monitor Agent — watches for alerts, investigates incidents web, search, file-read

The key insight: each agent has a bounded scope. A dev agent shouldn't deploy. A review agent shouldn't write code. This separation of concerns is what makes multi-agent systems safe and auditable at scale.

📘 Teaching Tip — Role Separation Is Governance: The exam treats agent role separation as a governance mechanism, not just a design pattern. When a scenario describes an agent that both writes and deploys code, the correct answer is usually to split it into two agents with different permission boundaries. Think of it like separation of duties in financial systems — no single agent should have end-to-end power.

2. Orchestration Patterns: Hierarchical, Sequential, Parallel, Competitive

Once you have multiple agents, you need a way to coordinate them. The exam defines four core orchestration patterns. Each maps to a specific GitHub-native mechanism.

Hierarchical (Parent/Child via agent tool)

A parent agent delegates sub-tasks to child agents using the agent tool. The parent defines the task, the child executes it, and the parent collects the result. This is the most flexible pattern — the parent can dynamically decide which child to invoke based on the task at hand.

  • When to use: Complex tasks that require specialized sub-tasks; dynamic delegation based on runtime context
  • GitHub mechanism: Agent profile includes tools: [agent]; parent references child agent IDs
  • Pros: Dynamic, flexible, perfect for divide-and-conquer
  • Cons: Parent becomes a potential single point of failure; parent must have tool access

Sequential (Pipeline via needs)

Agents execute in a fixed sequence, where each agent's output feeds the next. This is implemented via GitHub Actions needs — Job B waits for Job A to complete before starting.

  • When to use: Linear workflows where each step depends on the previous (plan → code → test → deploy)
  • GitHub mechanism: jobs: { job_a: {…}, job_b: { needs: [job_a], … } }
  • Pros: Predictable, easy to reason about, natural dependency model
  • Cons: Slow — total time is sum of all steps; failures block the entire pipeline

Parallel (via strategy.matrix)

Multiple agents execute the same task with different inputs or configurations simultaneously. This is the fan-out pattern, implemented via GitHub Actions strategy.matrix.

  • When to use: Running tests across multiple OS/version combinations; scanning many files concurrently
  • GitHub mechanism: strategy: { matrix: { os: [ubuntu, macos, windows] } }
  • Pros: Fast — total time equals the slowest parallel job; natural for CI/CD
  • Cons: Matrix sprawl (too many combinations); harder to collect and aggregate results

Competitive (Contest Pattern)

Multiple agents independently attempt the same task. The first to produce a valid result wins; other agents are cancelled. This is less common but powerful for latency-sensitive tasks.

  • When to use: When you need the fastest possible answer; when different approaches might yield different quality results
  • GitHub mechanism: Custom workflow with cancellation of redundant jobs; result comparison gate
  • Pros: Lowest latency; redundancy against individual agent failure
  • Cons: Resource waste (N-1 agents' work is discarded); result reconciliation complexity

⚠️ Exam Trap: The exam will give you a scenario and ask which orchestration pattern to use. If the scenario mentions dynamic delegation where a lead agent decides sub-tasks at runtime → hierarchical. If it's a fixed sequence of steps → sequential. If it's same task with different inputs → parallel/matrix. If it's multiple agents racing for the fastest result → competitive. Don't mix them up — each pattern has a distinct trigger condition.

4. Agent Isolation: Branches, Workflows, Permissions, Concurrency

When multiple agents operate in the same repository, you need isolation — agents must not interfere with each other's work. The exam identifies four isolation mechanisms:

Branch Isolation

Each agent works in its own branch. This is the simplest and most important isolation mechanism — changes are staged in separate branches until they're ready to merge.

  • Implement via: GitHub branch protection rules + agent workflow creates unique branch per task
  • Best practice: Use descriptive branch naming like agent/{agent-name}/{task-id}
  • Exam tip: Branch isolation prevents agents from overwriting each other's work in progress

Workflow Isolation

Each agent runs in its own GitHub Actions workflow (or job within a workflow). Workflow isolation prevents one agent's execution from affecting another's runtime environment.

  • Implement via: Separate workflow YAML files per agent or per task type
  • Best practice: Use separate runners or container environments per agent type
  • Exam tip: Workflow isolation is about runtime separation, not just code separation

Permission Isolation

Each agent has a unique set of permissions via its agent profile. Permissions define which tools, files, and actions the agent can access.

  • Implement via: .github/agents/*.agent.md — each agent has its own tools and permissions block
  • Best practice: Apply least-privilege: give each agent only the tools it needs for its role
  • Exam tip: A dev agent shouldn't have deploy permissions; a monitor agent shouldn't have write permissions

Concurrency Controls

GitHub Actions concurrency groups prevent multiple jobs or workflows from running simultaneously on the same resource.

  • Implement via: concurrency: key in workflow YAML with a group name
  • Best practice: Use cancel-in-progress: true to cancel older runs when a new one starts
  • Exam tip: Concurrency groups are essential for preventing race conditions in multi-agent deployments
# Example: Concurrency control for deployment agent name: Deploy Agent on: workflow_dispatch: inputs: environment: description: 'Target environment' required: true type: choice options: [staging, production] concurrency: group: deploy-${{ github.event.inputs.environment }} cancel-in-progress: true jobs: deploy: runs-on: ubuntu-latest environment: ${{ github.event.inputs.environment }} steps: - uses: actions/checkout@v4 - name: Deploy run: ./deploy.sh

5. Conflict Detection and Resolution

When multiple agents work concurrently, conflicts are inevitable. The exam tests your understanding of how GitHub-native features help detect and resolve these conflicts.

PR Reviews as Conflict Resolution

Pull requests are the primary conflict resolution mechanism. When two agents make changes to the same repository, their work is merged via PRs, which:

  • Surface merge conflicts that must be resolved before merging
  • Provide a review process for human or agent reviewers to validate changes
  • Require status checks (CI) to pass, ensuring conflicting changes don't break the build
  • Enable CODEOWNERS approval for files that need specialized review

Concurrency Groups for Conflict Prevention

As discussed above, concurrency groups prevent conflicting workflows from running at the same time. This is a preventive measure — it stops conflicts before they happen.

Branch Protection Rules

Branch protection rules enforce conflict-avoidance policies:

  • Require linear history — prevents merge commits and forces rebase, keeping history clean
  • Require up-to-date branches — forces agents to rebase on latest main before merging
  • Require status checks — ensures all evaluation gates pass before merge

📘 Teaching Tip — Conflict Resolution Flow: Think of conflict resolution as a three-stage pipeline: Prevent (concurrency groups, branch isolation) → Detect (PR merge conflict detection, CI failures) → Resolve (PR review, human mediation, rebase). The exam will ask you to identify which stage a given mechanism belongs to — make sure you can map each mechanism to its stage.

⚠️ Exam Trap: A question might present a scenario where two agents create conflicting changes and ask: "Which GitHub-native mechanism detects this conflict?" The answer is PR merge conflict detection, not concurrency groups. Concurrency groups prevent simultaneous execution conflicts; PR detection catches code-level conflicts. Know the difference.

6. System Observability: Attribution, Evidence, Cross-Job Handoffs

In a multi-agent system, you need to know who did what, when, and why. The exam defines three pillars of observability:

Attribution

Every action must be traceable to a specific agent. Attribution is achieved through:

  • Agent identity in workflow logs: Each agent's name and profile ID are logged with every action
  • Commit authorship: Git commits made by agents are attributed to the agent identity
  • PR authorship: PRs created by agents show the agent as the author

Evidence

Agents must produce evidence that they performed their tasks correctly. Evidence includes:

  • Workflow artifacts: Test reports, coverage summaries, build outputs
  • Session logs: Complete trace of agent actions, tool calls, and model responses
  • Check run results: Status checks that validate agent output

Cross-Job Handoffs

When one agent's output becomes another agent's input, the handoff must be observable:

  • Workflow outputs: Jobs produce outputs consumed by downstream jobs via needs
  • Artifact passing: Build artifacts are uploaded and downloaded between jobs
  • Workflow run URL: Each run has a unique URL that links all jobs together
  • Log correlation: Cross-referencing log entries across jobs via run ID and timestamps

🎓 Teaching Tip — Traceability Is Exam Gold: If an exam scenario describes a problem in a multi-agent system, the first step is always check attribution and logs. Who was the last agent to touch this file? What did the session log show? Traceability is the starting point for all debugging in multi-agent systems. The exam rewards this pattern relentlessly.

7. Failure Diagnosis and Recovery at Scale

When a multi-agent pipeline fails — and it will — you need a systematic approach to diagnosis and recovery. The exam expects you to know the diagnosis workflow and recovery strategies.

Diagnosis Workflow

Step Action Tool / Resource
1 Identify the failing job or agent Workflow run page, job status indicators
2 Examine session logs for error messages GitHub Actions log viewer, agent session logs
3 Check workflow artifacts for failure evidence Artifact download, test report viewer
4 Attribute the failure to an agent or action Log correlation, commit attribution
5 Determine the root cause category Diagnosis tree: config → instructions → tools → env → MCP
6 Apply the fix and re-run Tuning levers: instructions, tools, environment, MCP

Recovery Strategies

  • Retry with backoff: For transient failures (network timeouts, rate limits), configure automatic retries with exponential backoff in the workflow
  • Rollback and retry: For code-level failures, revert the agent's last changes and retry with corrected instructions
  • Fallback to human: For unresolvable failures, escalate to a human operator via workflow dispatch or issue creation
  • Partial replay: Re-run only the failed job (via workflow re-run), not the entire multi-agent pipeline
  • Checkpoint restart: Restore from a known-good checkpoint and resume execution from that point

Recovery at Scale

At scale, you need automated recovery. The exam highlights these patterns:

  • Self-healing agents: Configure agents to detect their own failures and attempt recovery before reporting failure
  • Circuit breaker pattern: After N consecutive failures, stop retrying and alert a human
  • Dead letter queue: Route unrecoverable failures to a queue for human review
  • Health monitoring: Regular heartbeat checks on agent availability and pipeline health

⚠️ Exam Trap: The exam may present a failure scenario and ask: "What's the best recovery strategy?" If the failure is transient (network blip, API rate limit), the answer is retry with backoff. If the failure is logic-based (wrong code, bad instructions), the answer is fix instructions and re-run. If the failure is unrecoverable (security violation, policy breach), the answer is escalate to human. Match the recovery strategy to the failure type.
