Memory, State, &
Execution

Agents can't do useful work if they forget everything between turns. This domain covers how agents remember, how state persists across sessions, and how execution continuity is maintained across tools and environments — including the critical concept of context drift detection and correction.

1. Why Memory Matters in Agentic AI

Welcome, student. Domain 3 is all about one question: how does an agent know what it's doing from one moment to the next? Without memory, every agent interaction starts from zero — no awareness of previous turns, no knowledge of repository facts, no recollection of user preferences. That's why memory is arguably the most important infrastructure component in any agent system.

The Microsoft Learn module "Memory, State, and Evaluation" and the GitHub Copilot Memory documentation define three distinct memory tiers. Understanding the difference between them — and when each is appropriate — is the core skill tested in this domain.

By the end of this lesson, you'll be able to:

  • Identify which memory type fits a given scenario
  • Explain how memory scoping, expiration, and pruning work
  • Describe Copilot Memory's repository-level and user-level mechanisms
  • Recognize context drift and choose the right correction strategy
  • Design for continuity across tool and environment boundaries

🎓 Teaching Tip — The Memory Mindset: Think of memory like a construction site. Short-term memory is the foreman's clipboard for today's tasks. Long-term memory is the blueprint repository and site rules. External durable artifacts are the actual building — the concrete and steel that remain after everyone goes home. Each serves a different purpose, and confusing them leads to architectural mistakes the exam loves to test.

2. The Three-Tier Memory Hierarchy

The exam defines three memory types in a strict hierarchy. Each has different persistence, scope, and access characteristics. Here's the visual model — study this diagram carefully:

Figure 3.1 — Three-Tier Memory Hierarchy: short-term (session), long-term (Copilot Memory), and external (durable artifacts). Information flows upward as it becomes more permanent.

Tier 1: Short-Term Memory (Session)

Short-term memory is the agent's working memory for the current conversation or session. It includes:

  • The current conversation history (user messages, agent responses, tool calls)
  • In-session decisions, intermediate results, and tentative plans
  • File contents the agent has read or edited during the session
  • Temporary state that exists only for the duration of the interaction

Short-term memory is volatile — it disappears when the session ends. Think of it as the agent's scratch pad. It's fast, local, and requires no special configuration.

Tier 2: Long-Term Memory (Copilot Memory)

Long-term memory — officially called Copilot Memory — persists across sessions. It stores two categories of information:

  • Repository-level facts: Key files, function signatures, documentation references, architectural patterns that the agent learns about your codebase
  • User-level preferences: Coding style preferences, testing conventions, frequently used libraries, personal workflow habits

Copilot Memory has a 28-day auto-deletion policy. Information that isn't reinforced within 28 days is automatically pruned. This prevents stale or outdated knowledge from accumulating.

Tier 3: External / Durable Artifacts

External durable artifacts are the permanent records of agent actions — the outputs that exist independently of any agent session:

  • Commits: Source code changes that are permanently recorded in Git history
  • Pull Requests: Structured change proposals with discussion history and reviews
  • Workflow logs: CI/CD execution records that capture every command and output
  • Issues and comments: Persistent communication threads about bugs, features, and tasks

These artifacts are the system of record — they don't expire, and they can be referenced by any agent or human at any time in the future.

⚠️ Exam Alert: The exam will test whether you know which memory tier applies in a given scenario. If the scenario says "the agent needs to remember a preference across sessions" → that's Copilot Memory (Tier 2). If it says "the agent needs a permanent record of a configuration change" → that's a durable artifact (Tier 3). Don't confuse session memory with long-term memory — session data is never persisted.

3. Memory Scoping, Expiration, and Pruning

Memory isn't just about storage — it's about relevance. An agent needs to know what information is relevant to the current task and what information should be discarded. The exam tests three memory management mechanisms:

Scoping

Memory scoping determines which information is visible to the agent during a given task. Effective scoping ensures the agent's context window isn't cluttered with irrelevant data. Scoping rules include:

  • Repository scope: Facts are scoped to the repository where they were learned
  • Task scope: Only memory entries tagged with relevant task labels are surfaced
  • Tool scope: Memory from specific tool interactions (e.g., search results) is scoped to that tool's domain

Expiration

Memory entries have time-based expiration. Copilot Memory automatically expires after 28 days if not reinforced. Short-term memory expires at the end of the session. Durable artifacts have no expiration — they persist indefinitely.

Pruning

When memory exceeds capacity (especially the context window), the agent must prune less relevant information. Pruning strategies include:

  • Recency-based: Keep the most recent N entries, discard older ones
  • Relevance-based: Keep entries that match the current task's semantic context
  • Priority-based: Keep high-priority entries (e.g., user preferences) over low-priority ones (e.g., intermediate calculations)

Reset Rules

Sometimes memory needs to be explicitly reset. Reset triggers include:

  • New task: Starting a fundamentally different task clears session context
  • Error recovery: After a critical failure, resetting memory prevents cascading errors
  • User request: The user can explicitly request a memory reset
  • Context drift detection: The system detects the agent is operating on stale assumptions and triggers a reset

📘 Teaching Tip — The "Freshness" Concept: The exam is big on memory freshness. Imagine an agent that memorized your database schema 30 days ago. If the schema changed last week, that memory is stale. The 28-day expiration window in Copilot Memory is designed to prevent exactly this problem. When you see a question about stale information, the answer always involves expiration, pruning, or refreshing from durable artifacts.

4. Copilot Memory — Repository Facts and User Preferences

Copilot Memory is the GitHub-specific implementation of long-term memory. It's the mechanism that allows GitHub Copilot (both the IDE extension and the CLI agent) to retain knowledge across sessions. Let me give you the exam-relevant details.

Repository-Level Facts

When an agent works in a repository, Copilot Memory learns facts about that repository:

  • Key file locations: Where the main entry point, configuration files, and test suites live
  • Architectural patterns: How the codebase is organized (e.g., MVC, microservices, monorepo)
  • API contracts: Function signatures, return types, and expected inputs
  • Code conventions: Naming conventions, file structure, import patterns
  • Documentation references: Key README sections, wiki pages, architectural decision records

These facts are repository-scoped — an agent working in repo A cannot access facts learned in repo B. This is a security and isolation mechanism.

User-Level Preferences

Copilot Memory also stores preferences tied to the user (not the repository):

  • Coding style preferences: Tabs vs spaces, naming conventions, comment styles
  • Testing preferences: Preferred test framework, coverage requirements, mocking strategy
  • Tooling preferences: Which linter, formatter, or build system the user prefers
  • Workflow habits: Whether the user prefers PRs or direct commits, rebase or merge

28-Day Auto-Deletion

This is the single most important Copilot Memory detail for the exam. All memories have a 28-day time-to-live (TTL):

  • Every memory entry has a creation timestamp
  • After 28 days, the entry is automatically deleted
  • Accessing or reinforcing a memory resets its TTL
  • Deleted memories cannot be recovered — they must be re-learned

This design prevents knowledge drift — the risk that the agent acts on outdated information. It also means agents must regularly interact with the repository to keep their memory fresh.

⚠️ Critical Exam Detail: The 28-day TTL applies to Copilot Memory only, not to durable artifacts. A commit you made 6 months ago is still accessible via Git history. But a preference Copilot learned about your coding style 6 months ago will have been deleted after 28 days if not reinforced. The exam WILL ask: "An agent needs to remember a fact from last year." The answer is NOT Copilot Memory — it's a durable artifact (a commit, a PR, a file in the repo).

5. State Persistence via Durable Artifacts

When an agent needs to preserve state permanently, it uses durable artifacts. These are the concrete outputs of agent activity that survive session termination and agent restarts.

Types of Durable Artifacts

Artifact Persistence Mechanism Example Use Case
Commits Git history (immutable after push) Agent writes a fix, commits it, and pushes — the change is permanent
Pull Requests GitHub database (persistent) Agent creates a PR with a proposed change — the thread and diff persist
Workflow Logs GitHub Actions log storage (90-day retention) Agent runs a CI pipeline — every output and error is logged
Issues & Comments GitHub database (persistent) Agent files a bug report with diagnostic information
Releases & Tags GitHub releases (persistent) Agent cuts a release with changelog and binary artifacts

Artifact as Memory

Durable artifacts serve as an external memory store. When an agent starts a new session, it can read durable artifacts to reconstruct context. For example:

  • Reading a previous PR's description to understand why a change was made
  • Inspecting workflow logs to diagnose a previous CI failure
  • Reviewing issues to understand the history of a bug

This is the pattern the exam expects: short-term memory for the current session, Copilot Memory for cross-session preferences and facts, and durable artifacts for permanent state that must survive everything.

🎓 Teaching Tip — The "Write It Down" Rule: If you want an agent to remember something permanently, the answer is always: write it to a durable artifact. Commit a config file. Open an issue. Post a comment. Create a PR. If the memory exists only in the agent's short-term or long-term memory, it's temporary. The exam loves this distinction.

6. Context Drift: Detection, Correction, and Re-Planning

Context drift is what happens when the agent's internal model of the world diverges from reality. It's one of the most dangerous failure modes in agentic systems because it's silent — the agent doesn't know it's operating on outdated assumptions.

What Causes Context Drift?

  • Stale memory: The agent remembers a file structure that has since changed
  • External state changes: An environment variable, API response, or database state changed without the agent noticing
  • Concurrent modifications: Another agent or human changed files while this agent was working
  • Time-sensitive information: The agent is using timing-dependent data (e.g., rate limits, token expiration) that is no longer valid
  • Erroneous assumptions: The agent inferred something about the system that was never true

Detection Strategies

The exam expects you to know how context drift is detected:

  • Re-read confirmation: Before making a change, the agent re-reads the file to verify its mental model matches reality
  • Timestamp checking: Comparing the agent's last-known state timestamp with the actual file modification timestamp
  • Consistency validation: Running validation checks (e.g., parsing config files) to confirm expected structure
  • Diff comparison: Comparing the expected file content with the actual content before editing

Correction and Re-Planning

Once context drift is detected, the agent must:

  1. Flag the drift: Log the discrepancy for auditability
  2. Refresh context: Re-read the relevant files, re-query the environment, and rebuild the mental model
  3. Re-plan: Re-evaluate the plan against the corrected context — the original plan may no longer be valid
  4. Escalate if needed: If the drift is severe enough to invalidate the entire plan, escalate to a human

Preventing Context Drift

Prevention is better than correction. Best practices include:

  • Validate before each write: Re-read the target file immediately before editing
  • Use transactions: Group related changes so partial failures don't leave inconsistent state
  • Short task durations: Shorter tasks are less likely to experience drift than long-running tasks
  • Environment checks: Verify environment variables, service health, and auth tokens at the start of each task

⚠️ Exam Trap: A common wrong answer on context drift questions is "just retry the operation." Retrying without refreshing context will fail the same way — the agent will use the same stale assumptions. The correct response is always re-read → re-validate → re-plan, not just retry. If you see a question where an agent keeps failing the same operation, look for the answer that includes re-reading the current state.

7. Continuity Across Tools and Environments

Modern agent workflows span multiple tools and environments. An agent might start work in the IDE, continue in the CLI, and execute in CI — all within the same task. The exam tests whether you understand how continuity is maintained across these boundaries.

The Continuity Challenge

Each environment has different capabilities and limitations:

Environment Memory Available Continuity Mechanism
IDE (VS Code) Short-term + Copilot Memory Session persists as long as editor is open; Copilot Memory bridges sessions
CLI (gh copilot) Short-term (CLI session) + Copilot Memory CLI sessions are independent; shared Copilot Memory provides continuity
CI (GitHub Actions) Short-term (workflow run) only No cross-run memory; relies on durable artifacts (logs, artifacts, git state)
Codespaces Short-term + Copilot Memory Workspace persists while Codespace is running; Copilot Memory bridges stops/starts

Continuity Patterns

The exam expects you to know how to design for continuity:

  • Checkpoint artifacts: Write intermediate state to files (JSON, YAML) that the next environment can read
  • Shared configuration: Use .github/copilot-instructions.md and .github/instructions/*.instructions.md so all environments have the same behavioral guidelines
  • Token-based identity: Use GitHub tokens (GitHub App installation tokens or PATs) so the agent can authenticate in any environment
  • State files in the repo: Store agent progress state in a well-known path like .github/agent-state/
# Example: Continuity via checkpoint file # Written by IDE agent, read by CI agent .github/agent-state/task-42-checkpoint.json: { "task_id": "42", "phase": "code-review", "files_scanned": [ "src/auth/login.ts", "src/auth/register.ts" ], "findings": [ { "severity": "high", "file": "src/auth/login.ts", "line": 45, "description": "SQL injection vulnerability in raw query" } ], "next_environment": "ci", "continuation_url": "https://github.com/org/repo/actions/runs/12345" }

📘 Teaching Tip — Environment Handoff: On the exam, questions about environment handoff are really testing durable artifact usage. The agent in environment A writes its state to a file (commit or artifact), and the agent in environment B reads that file to continue. There's no magic "environment bridge" — it's always: write state, transfer via artifact, read state. Remember this pattern and you'll get every continuity question right.
