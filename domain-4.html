Domain 4 · Weight: 15-20%

Evaluation, Error Analysis,
& Tuning

You've built your agent. Now how do you know if it's working? Domain 4 is about the feedback cycle — defining success criteria, detecting failures, diagnosing root causes, and iteratively tuning agent behavior. This is where theory meets practice on the exam.

1. The Evaluation Mindset

Welcome to Domain 4. If Domain 3 was about how agents remember, Domain 4 is about how agents improve. The core loop is simple: Define criteria → Execute → Measure → Analyze → Tune → Repeat. This is the Plan-Act-Evaluate (PAE) cycle that underpins all agentic systems, and the exam expects you to understand every step.

In this lesson, I'll teach you:

  • How to define meaningful success criteria that an agent or CI pipeline can verify
  • Which automated scanning tools are available and what each one detects
  • How to do root cause analysis when an agent fails — with a structured decision framework
  • The four tuning levers: instructions, tools, environment, and MCP
  • How session logs, workflow artifacts, and PR reviews serve as evaluation methods

📘 Teaching Tip — Evaluation Is Not Optional: The exam treats evaluation as a first-class concern, not an afterthought. Every agent workflow should have explicit evaluation checkpoints. If a question presents an agent workflow without any evaluation step, the implied correct answer is always: "add an evaluation gate." This is similar to how you'd never deploy code without tests — you should never deploy an agent configuration without evaluation.

3. Defining Success Criteria and Evaluation Signals

Every agent task needs explicit success criteria before the agent starts. The exam distinguishes between weak and strong evaluation signals.

Weak vs Strong Signals

Type Example Problem
❌ Weak "The PR looks good" Subjective, unmeasurable, depends on reviewer opinion
❌ Weak "Tests pass" Tests could be incomplete or not cover the agent's changes
✅ Strong "All existing + new tests pass; coverage ≥ 80%; no lint warnings" Quantitative, verifiable, multi-dimensional
✅ Strong "PR has exactly 3 files changed (2 source + 1 test); all status checks green" Scope-bound, automated, specific

Evaluation Signal Types

The exam defines several categories of evaluation signals:

  • Functional signals: Does the output do what it's supposed to? (Tests pass, code compiles, API returns expected status)
  • Quality signals: Is the output well-made? (Code coverage, lint score, performance benchmarks, accessibility checks)
  • Compliance signals: Does the output follow governance rules? (No secrets committed, follows team conventions, meets security policies)
  • Process signals: Was the workflow followed correctly? (Plan was reviewed, approvals obtained, audit trail complete)

🎓 Teaching Tip — The "So What?" Test: For every success criterion you define, ask: "If this criterion passes, does that actually mean the agent succeeded?" If "tests pass" is your only criterion, the agent could delete all tests (making them pass vacuously) or write trivial tests that assert nothing. Strong criteria are specific, quantitative, and hard to game.

4. Automated Scanning Tools for Evaluation

GitHub provides several automated scanning tools that serve as evaluation signals for agent outputs. The exam expects you to know what each tool does and when to use it.

Tool What It Scans What It Detects Trigger
CodeQL Source code (all pushed branches and PRs) Security vulnerabilities, code quality issues, logic errors, injection flaws, XSS, SQLi, path traversal CI workflow or push/PR trigger in .github/workflows/codeql.yml
Dependabot Dependency manifests (package.json, requirements.txt, Gemfile, etc.) Known vulnerabilities in dependencies (CVEs), outdated packages, version incompatibilities Automated — scans on push and on schedule; auto-creates PRs for fixes
Secret Scanning All repository content (commits, PRs, issues, descriptions) Hard-coded secrets: API keys, tokens, passwords, certificates, private keys, connection strings Real-time — detects on push; alerts security team; can block push with push protection

How These Fit Into Evaluation

Each tool produces evaluation signals that answer specific questions about agent output:

  • CodeQL answers: "Did the agent introduce any security vulnerabilities?"
  • Dependabot answers: "Did the agent introduce any vulnerable dependencies?"
  • Secret Scanning answers: "Did the agent accidentally expose any credentials?"

These tools should be part of every agent workflow as automated evaluation gates. If a PR from an agent triggers a CodeQL alert, the evaluation fails — the agent must fix the issue before the PR can merge.

⚠️ Exam Trap: The exam will test whether you know which scanning tool to use for a given scenario. If the scenario mentions a vulnerable library → that's Dependabot. If it mentions hard-coded API keys → that's Secret Scanning. If it mentions SQL injection in custom code → that's CodeQL. Don't confuse them — each tool has a distinct purpose.

5. Root Cause Analysis: Symptom → Root Cause → Fix

When an agent fails, the exam expects you to trace the failure from symptom back to root cause, then choose the appropriate fix. Here's the structured decision framework:

Symptom Likely Root Cause Fix
Agent produces incorrect code (wrong logic, wrong API calls) Instructions too vague — agent lacks specific behavioral guidelines for the task Add detailed task-specific instructions in .github/instructions/*.instructions.md or improve the prompt template
Agent can't find the right files or functions Missing context or search scope too narrow — agent doesn't know the codebase structure Provide file path hints, expand search scope, or add Copilot Memory entries with key file locations
Agent tries to use a tool it doesn't have access to Tool misconfiguration in agent profile — the tools list in .github/agents/*.agent.md doesn't include the required tool Add the missing tool to the agent's profile's tools list, or reassign the task to an agent with the right permissions
Agent's MCP calls fail MCP server misconfiguration — server not defined in .mcp.json, wrong host/port, or authentication issues Verify MCP server config in .mcp.json or .github/mcp.json, check server is running and reachable
Agent makes changes to wrong files or scope Context drift or insufficient scoping — agent's mental model of which files are in scope is wrong Re-read files before editing; add explicit scope constraints to task instructions; use file path patterns in tool config
Agent's PR gets rejected in review Poor output quality — agent didn't follow team conventions, missing tests, no documentation Add PR template requirements; include quality checks in the evaluation criteria; tune agent instructions for the expected output format
Agent silently fails (no output, no error) No error handling or logging — agent encounters an error but doesn't surface it Implement error handling in the agent workflow; configure hooks at .github/hooks/*.json for lifecycle monitoring; add logging and alerting
Agent keeps repeating the same mistake Insufficient feedback loop — evaluation signals are too weak or don't reach the agent's tuning mechanism Strengthen evaluation criteria; add automated scanning gates; implement a formal iterative tuning process (PAE loop)
Agent's output violates security policies Lack of guardrails or insufficient autonomy level — agent has too much freedom without governance Reduce autonomy level; add CodeQL/Secret Scanning gates; enforce branch protection rules; add human review requirement
Agent uses the wrong tool for the task Autonomy mismatch or misconfigured least privilege — agent tool set doesn't match the task requirements Audit the agent's tools list; apply least-privilege principle; add tool-use instructions in the agent profile or prompt

📘 Teaching Tip — The Diagnosis Tree: When the exam presents a failure scenario, always start by asking: "Is the agent's configuration correct?" If yes, ask: "Are the instructions complete enough?" If yes, ask: "Is the evaluation catching the right signals?" Work through this tree — the exam's wrong answers typically skip a level and go directly to a complicated fix when the real fix is simple configuration.

6. The Four Tuning Levers: Instructions, Tools, Environment, MCP

When evaluation reveals that an agent isn't performing as expected, you have four tuning levers to adjust. The exam expects you to know which lever to use for each type of problem.

Lever 1: Instructions

What it controls: The agent's understanding of what to do and how to behave.

When to adjust: When the agent produces incorrect or low-quality output because it doesn't have enough context, doesn't understand the expected output format, or follows wrong conventions.

How to adjust:

  • Update .github/copilot-instructions.md for repo-wide behavior
  • Create or modify .github/instructions/*.instructions.md for task-specific guidance
  • Improve prompt templates in .github/prompts/*.prompt.md
  • Add explicit examples of desired output (few-shot prompting)

Lever 2: Tools

What it controls: Which actions the agent is allowed to take.

When to adjust: When the agent tries to use tools it doesn't have, or when it has tools it shouldn't use (privilege creep). Also when the agent's tool permissions don't match the task's requirements.

How to adjust:

  • Update the tools list in .github/agents/*.agent.md
  • Add tool-specific constraints (restricted commands, allowed directories)
  • Enable or disable the web tool for external HTTP access
  • Add or remove the agent tool for sub-agent delegation

Lever 3: Environment

What it controls: The execution context where the agent runs — including secrets, environment variables, runtime versions, and workspace layout.

When to adjust: When the agent fails due to missing dependencies, wrong runtime versions, missing authentication tokens, or incorrect workspace setup.

How to adjust:

  • Configure .github/workflows/copilot-setup-steps.yml for CI environment
  • Set environment variables and secrets in GitHub Actions or Codespaces config
  • Define target in the agent profile (e.g., github-actions, codespaces, cli)
  • Configure environment protection rules for deployment gates

Lever 4: MCP

What it controls: Which external services the agent can access via the Model Context Protocol.

When to adjust: When the agent needs access to external data sources, APIs, or specialized services that aren't available through its built-in tools.

How to adjust:

  • Add or modify MCP server definitions in .mcp.json, .github/mcp.json, or .vscode/mcp.json
  • Add MCP server names to the agent profile's mcp-servers list
  • Configure the MCP allow list for security control
  • Register custom MCP servers in the MCP registry

🎓 Teaching Tip — The Lever Priority: Start with instructions (cheapest, lowest risk), then tools (still low risk), then environment (medium risk), then MCP (highest risk, widest blast radius). The exam will test whether you choose the minimal effective change. Don't reconfigure MCP servers when a better instruction file would fix the problem.

7. Evaluation Methods: Session Logs, Workflow Artifacts, PR Reviews

Beyond automated scanning tools, the exam recognizes three primary methods for evaluating agent performance. Each provides different signals at different stages of the agent workflow.

Session Logs

Session logs capture the complete record of an agent's interaction: every prompt, every tool call, every output, every error. They are the ground truth for understanding what the agent did and why.

  • Use for: Post-mortem analysis, debugging failures, auditing agent decisions
  • Contains: Timestamped sequence of events, tool inputs and outputs, model responses, error messages
  • Storage: GitHub Actions workflow logs (90-day retention), local log files, or external log aggregation
  • Exam tip: If an agent produces unexpected output, the first diagnostic step is always: check the session logs

Workflow Artifacts

Workflow artifacts are the persistent outputs generated during agent execution — test reports, coverage summaries, build outputs, scanned file lists, generated documentation.

  • Use for: Verifying that the agent's output meets quality and completeness criteria
  • Contains: Build artifacts, test results, coverage reports, lint output, security scan results
  • Storage: GitHub Actions artifact storage, committed files in the repository, or external blob storage
  • Exam tip: Workflow artifacts serve as durable evidence that evaluation happened — they can be inspected by humans or other agents later

PR Review

PR review is the human evaluation gate. It's the final check before an agent's changes enter the main codebase.

  • Use for: Catching issues that automated tools miss (design concerns, code clarity, business logic correctness)
  • Mechanism: Required reviewers (CODEOWNERS), review approval requirements, PR templates, change request workflows
  • Integration: PR review is the bridge between automated evaluation and human judgment
  • Exam tip: PR review is not a substitute for automated evaluation — it's a complement. Automated checks run first; humans review the results and make the final call
# Evaluation workflow: Combining all three methods name: Agent Evaluation on: pull_request: types: [opened, synchronize] jobs: # 1. Automated scanning (CodeQL, Dependabot, Secret Scanning) security-scan: uses: ./.github/workflows/security-scan.yml # 2. Build and test (produces workflow artifacts) build-test: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - run: npm ci - run: npm test -- --coverage - run: npm run lint - uses: actions/upload-artifact@v4 with: name: test-results path: coverage/ # 3. Session logs are automatically captured by GitHub Actions # 4. Human PR review is required by branch protection rules

⚠️ Exam Trap: A question might ask: "What's the best evaluation method for catching business logic errors?" The answer is PR review, not CodeQL or Dependabot. Automated tools catch technical errors (vulnerabilities, CVEs, secrets). Only a human reviewer can catch business logic errors — an agent might produce perfectly valid code that does the wrong thing. Know the boundary between automated and human evaluation.

8. Putting It All Together: A Tuning Workflow

Here's a complete example of how the PAE loop works in practice. The agent fails evaluation, we analyze the root cause, apply a tuning change, and re-evaluate:

# Scenario: Agent generates test files without assertions # Step 1: Define success criteria # Criteria: Every generated test file must have at least one assertion # Measurement: Count assertions per test file via AST analysis # Step 2: Agent executes → generates empty test files → FAIL # Step 3: Analyze root cause # Symptom: Tests missing assertions # Root cause: Instructions don't specify assertion requirements # Correction: Add assertion requirement to instructions # Step 4: Apply tuning change # File: .github/instructions/test-generation.instructions.md # Before: # "Generate unit tests for the given functions." # After: # "Generate unit tests for the given functions. # Each test file MUST include assertions using assert() or expect(). # Every test function must have at least one assertion. # Include edge case tests (empty input, null, boundary values)." # Step 5: Re-run evaluation # Agent regenerates tests → assertions present → PASS # Step 6: Document the lesson learned # Add to .github/copilot-instructions.md: # "All agent-generated test files must contain assertions. # Validation: Automated assertion-count check runs after test generation."

📘 Teaching Tip — The Iteration Mindset: The exam wants to see that you understand evaluation as a continuous process, not a one-time check. Each evaluation cycle produces insights that feed into the next cycle. Over time, your instructions get better, your tool configurations get tighter, and your evaluation signals get stronger. This is the Plan-Act-Evaluate loop operating at the meta level — evaluating your evaluation.
